# Как работает доступность в Cairn (DOM-зеркало над canvas)

Простыми словами: как «немой» canvas начинает вести себя как настоящие нативные элементы — с фокусом, клавиатурой и скринридерами — и как DOM и canvas при этом общаются.

---

## Проблема: canvas — чёрный ящик

Мы рисуем весь интерфейс пикселями на одном `<canvas>`. Для глаза это кнопки и поля. Но для **скринридера, клавиатуры и ОС это один сплошной прямоугольник** — там нет ни кнопок, ни ролей, ни имён. Tab некуда ставить фокус, VoiceOver нечего озвучить. Нарисовать кнопку ≠ сделать кнопку.

## Решение: прозрачный «пульт» из настоящих элементов

Поверх canvas мы держим **скрытый слой из настоящих DOM-элементов** (`<button>`, `role`, `aria-*`, `tabindex`) — по одному на каждый интерактивный узел, спозиционированный ровно над своим местом на картинке. Слой прозрачный и `pointer-events:none`.

**Аналогия:** canvas — красивая картинка (шкурка), а скрытый DOM — прозрачный пульт из реальных кнопок поверх неё. Браузер и ОС нажимают пульт (нативно, со всей доступностью), пульт дёргает наши функции, а мы перерисовываем картинку. Так работает и Flutter в вебе.

Важно: **нативность у нас настоящая, а не имитация** — под капотом реальные элементы, поэтому фокус, Tab, Enter/Space, скринридеры и даже нативный сабмит формы работают сами.

---

## Как они общаются — два направления

### Направление 1: Canvas → DOM («опиши, что на экране»)

Каждый компонент вешает на свой узел платформо-независимый дескриптор (ноль DOM):

```ts
instance.semantics = {
  role: 'button', label: 'Отправить', disabled: false,
  onActivate: () => onClick(),            // что делать при активации
  onFocus: (kb) => setFocusVisible(kb),   // kb = фокус пришёл с клавиатуры?
  onBlur:  () => setFocusVisible(false),
};
```

Каждый кадр (после layout):
1. `collectSemantics(root)` обходит дерево инстансов (DFS = порядок Tab) → плоский список `SemanticsNodeData[]`. У каждого: **стабильный id**, **абсолютный прямоугольник** (из смещений layout), aria-поля и **ссылки на колбэки**.
2. `host.a11y.sync(nodes)` отдаёт список мосту.
3. `WebAccessibilityBridge` **сверяет (diff)** скрытый оверлей: создаёт/обновляет/удаляет DOM-элементы по id, ставит `role`/`aria-*`/`tabindex` и позицию из rect.

Это односторонний поток «состояние экрана → зеркало», пересобираемый каждый кадр.

### Направление 2: DOM → Canvas («пользователь что-то сделал»)

1. Пользователь жмёт Tab / Enter / активирует через скринридер → **настоящий `<button>`** стреляет нативным `click`/`focus`/`blur`.
2. Слушатели моста зовут колбэки узла: `click → onActivate`, `focus → onFocus(keyboard?)`, `blur → onBlur`.
3. Эти колбэки — **функции самого компонента**. Они меняют сигналы/зовут хендлеры.
4. Сигнал изменился → реактивность → планируется кадр → **canvas перерисовывается** (focus-ring, новое состояние).

### Указатель идёт отдельно

Мышь и тач по **canvas** обрабатывает canvas-пайплайн (hover/press + клик → тот же `onActivate`). DOM-оверлей `pointer-events:none`, мышь его не трогает. **Клавиатура/фокус/скринридер** идут через DOM-зеркало. Оба пути сходятся в одни действия → двойной активации нет. Когда a11y-мост есть, нативный фокус — источник истины, а свой canvas-Tab-менеджер отключается.

## Схема цикла

```
   компонент:  instance.semantics = { role, label, onActivate, onFocus, ... }
        │
        ▼  (каждый кадр)
   collectSemantics ──► SemanticsNodeData[] ──► host.a11y.sync()
                                                     │
                                                     ▼
                                     WebAccessibilityBridge
                                     скрытый <div> с реальными
                                     <button role aria-* tabindex>,
                                     спозиционированными по rect
                                                     │
                              Tab / Enter / VoiceOver / фокус
                                                     │
                                     нативное click / focus / blur
                                                     ▼
        onActivate / onFocus(kb) / onBlur  ◄──  слушатели моста
                    │
                    ▼
        сигналы меняются → scheduleFrame → canvas перерисован
```

---

## Что связывает две стороны

- **id (WeakMap)** — линкует инстанс на canvas ↔ его DOM-элемент между кадрами. Diff стабилен, и **фокус не слетает**.
  > Реальный баг, который мы поймали: мост пере-аппендил каждый элемент каждый кадр ради порядка Tab, а `appendChild` перемещает узел → **сбрасывает фокус**, если он на нём. А синк идёт каждый кадр → фокус/клавиатуру постоянно уводило. Починка: переупорядочивать только когда порядок реально изменился, с сохранением фокуса. Любой пофрейм-реконсилер DOM-зеркала обязан не двигать сфокусированный элемент.
- **rect** — DOM-элемент лежит ровно над своим canvas-узлом (в CSS-пикселях), чтобы подсветка фокуса скринридера, тач-исследование и нативный scroll-to-focus попадали в нужное место.
- **колбэки в дескрипторе** — единственный «провод» из DOM обратно в компонент; никакого id-роутинга не нужно.

---

## Тонкость: focus-visible (кольцо только с клавиатуры)

Мост запоминает **модальность** последнего ввода: `keydown → 'keyboard'`, `pointerdown → 'pointer'`. При фокусе он зовёт `onFocus(modality === 'keyboard')`. Поэтому кольцо фокуса рисуется **только при Tab**, а при клике мышью — нет, как у нативных элементов. Само кольцо — первоклассный стиль `outline` (обводка снаружи бокса), а не тень (тень в нашем движке не даёт чистого кольца).

## Проверка (и грабли Playwright)

- `browser_snapshot` показывает **настоящее a11y-дерево** (роли/имена/состояния) — так проверяем, что скринридер увидит `button "Отправить"`, `checkbox [checked]` и т.п.
- Грабля: **Playwright в headless не диспатчит focus-события на программный `.focus()`/Tab, когда окно страницы не в системном фокусе** (`activeElement` меняется, а события `focus` нет). Поэтому focus-ring не проверить через `.focus()`+скриншот — диспатчим событие напрямую (`el.dispatchEvent(new FocusEvent('focus'))` после `keydown`) и/или полагаемся на unit-тесты.

## Итог одним абзацем

Доступность в Cairn — это прозрачный слой настоящих DOM-элементов поверх canvas. Каждый кадр canvas «рассказывает» слою, что на экране (роли, имена, позиции, состояния), а слой — благодаря тому, что это реальные элементы, — даёт нативные фокус, клавиатуру и скринридер бесплатно; когда пользователь взаимодействует, нативное событие дёргает колбэк компонента, тот меняет состояние, и canvas перерисовывается. Пиксели — для глаз, DOM-зеркало — для клавиатуры и ассистивных технологий, а связывают их стабильные id, прямоугольники и колбэки.
