### Task 1: `animateSpring` physics driver (@cairn/runtime)
- Files: new `packages/runtime/src/spring.ts` (`animateSpring`, `SpringOptions`), export from `packages/runtime/src/index.ts`. Test `packages/runtime/test/spring.test.ts`.
- Semi-implicit Euler, dt=(now-last)/1000 clamped ≤ 1/30. `a = (-k*(x-to) - c*v)/m; v += a*dt; x += v*dt`. Settle `|x-to|<restDelta && |v|<restSpeed` → onUpdate(to)+onDone. Defaults stiffness 170, damping 26, mass 1, restDelta 0.01, restSpeed 0.05. Returns `{ cancel, velocity: () => v }`. Uses `useHost().scheduler.requestFrame`; `onCleanup(cancel)`.
- TDD (fake manual-clock host — reuse the pattern from `packages/runtime/test/animate.test.ts`): feeding successive timestamps moves x from→to and settles (final onUpdate = to, onDone once); mid-flight `velocity()` ≠ 0; underdamped (low damping) overshoots past `to` at some tick; `initialVelocity` shifts early trajectory; cancel stops further ticks.

### Task 2: spring transitions + velocity carry-over
- Files: `packages/style/src/style.ts` (`TransitionConfig.spring?: { stiffness?; damping?; mass? }`), `packages/primitives/src/transitions.ts` (use spring path when configured), test `packages/primitives/test/transitions-spring.test.ts`.
- In `createStyleTransitions`, per prop: if the matching `TransitionConfig` has `spring`, drive progress 0→1 via `animateSpring({ from: 0, to: 1, ...spring, initialVelocity, onUpdate: (p) => set(interpolateValue(fromVal, toVal, p)), onDone: () => set(toVal) })`. Store the returned handle per prop. On retarget (target changed), read `handle.velocity()` as `initialVelocity`, cancel it, start a new spring from `fromVal = current displayed value`, `toVal = new target`, `from: 0`. Time-based configs keep the existing `animate` path. Track handles separately (a `Record<Anim, {cancel, velocity} | undefined>`).
- TDD (fake clock): a `{ opacity, transition: { spring: {} } }` moves toward target over ticks and settles at target; retarget mid-flight — assert the new spring was started with a non-zero `initialVelocity` (spy/observe: after retargeting mid-motion, the value continues past/toward without resetting velocity to 0 — a practical assertion: capture values, retarget at tick N, confirm the immediately-next value moved by more than a from-rest spring would, OR assert velocity() of the prior handle was read). Keep it deterministic; if hard to assert continuity precisely, at least assert: spring path reaches target and a retarget doesn't throw + continues to the new target.
